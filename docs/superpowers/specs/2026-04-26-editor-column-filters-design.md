# 文本编辑表格列筛选设计

## 背景

控制面板的“文本编辑”页面已经提供缓存文本表格、全局搜索、排序、列显隐、列顺序调整、列宽调整、复制粘贴、导入导出和行编辑。当前 `/api/translations` 只支持一个全局搜索参数，无法像 Excel 一样在列标题上逐列筛选，也无法先按场景缩小范围再继续按组件类型筛选。

缓存行已经包含适合筛选的字段：原文、译文、目标语言、场景、层级、组件、替换字体、服务商、模型、创建时间和更新时间。本次功能只改变查询和表格交互，不改变缓存存储格式。

## 目标

- 每一列的列标题都提供筛选入口。
- 支持多列叠加筛选，筛选条件之间使用 `AND` 语义。
- 支持逐层筛选：例如先选某个场景，再打开组件列时，只展示当前场景范围内存在的组件类型。
- 保留现有全局搜索、排序、分页、列显隐、列顺序、列宽和单元格编辑能力。
- 控制面板继续保持中文界面和本机回环访问，不新增外部服务。

## 方案

采用“服务端列筛选 + 列头下拉菜单”。

后端扩展 `TranslationCacheQuery`，新增按列筛选条件集合。`LocalHttpServer` 从 `/api/translations` 查询参数解析筛选条件，并传给缓存查询实现。SQLite、Memory、Disk 缓存使用同一语义：全局搜索和所有列筛选条件共同生效，排序和分页在筛选后执行。

前端在 `ControlPanelHtml.cs` 的表格列定义中为每列保留数据库列名和显示字段名。表头标题右侧加入筛选按钮。点击按钮打开该列菜单，菜单包含列内搜索、多选值列表、全选、清空和应用。已筛选列在表头显示状态标记。

为了让逐层筛选准确，新增筛选值接口，例如 `/api/translations/filter-options?column=scene_name`。前端请求该接口时会带上当前其他列筛选和全局搜索，后端返回该列在当前范围内的可选值及计数。请求某一列的可选值时，不应用该列自己已有的筛选，这样用户能重新调整该列选择。

## 查询语义

- 全局搜索继续匹配原文、译文、场景、层级、组件和替换字体。
- 列筛选使用精确值匹配，空值以专门的空值标记传输并匹配 `NULL` 或空字符串。
- 多列筛选为 `AND`：行必须满足所有已启用列筛选。
- 同一列内多选为 `OR`：行的该列值匹配任一已选值即可。
- 排序、分页和总数基于筛选后的结果。

## 前端交互

- 列标题仍可点击排序。
- 筛选按钮单独处理点击事件，不触发排序。
- 筛选菜单靠近当前列标题显示，并受表格容器宽度约束，避免溢出屏幕。
- 筛选值列表支持列内搜索，长文本以省略号显示并保留完整 `title`。
- “应用”后刷新表格并重置分页偏移。
- “清空”只清空当前列筛选；表格工具区可增加“清空筛选”按钮，用于一次清除所有列筛选。
- 表格底部消息显示当前筛选状态，例如“共 128 条，当前显示 100 条，已筛选 2 列”。

## 数据和兼容性

不修改 SQLite 表结构，不迁移现有缓存数据。筛选只读取现有列。导入、导出、更新和删除接口保持原有数据结构。

列名白名单由后端维护，避免把任意查询参数拼接进 SQL。SQLite 查询使用参数化 `IN` 条件；Memory 和 Disk 查询使用相同的列取值函数进行过滤。

## 测试

- 单列筛选：按场景筛选后只返回该场景的行。
- 多层筛选：按场景筛选后，再按组件筛选，只返回同时满足两列的行。
- 同列多选：选择多个组件类型时返回任一组件类型的行。
- 空值筛选：能筛出场景、层级、组件或替换字体为空的行。
- 全局搜索与列筛选叠加：两者同时生效。
- 排序和分页在筛选后仍正确。
- 列头筛选按钮不触发现有排序。
- 已筛选列隐藏再显示后，筛选状态仍保留。
